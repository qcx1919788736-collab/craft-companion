## 核心问题

AI 协作创作面临一个根本性的矛盾：

> **我们想让 AI"聪明地完成任务"，但 AI 的"聪明"往往表现为"过度自主"和"不可预测"。**

具体表现为：

| 问题 | 表现 | 根因 |
|------|------|------|
| **AI 过度自主** | 执行顺序不一致，有时跳过步骤 | 描述型路由让 AI 自己判断"该做什么" |
| **上下文泄露** | 不该看到的内容影响了创作 | 文件级隔离不足 |
| **AI 偷懒** | 跳过关键步骤（如自查）直接交稿 | 关键步骤是"可选"而非"必选" |
| **状态丢失** | AI 中断后需要重新说明上下文 | 状态存在于 AI 上下文而非外部文件 |

**核心洞察**：用"更好的指令"来解决这些问题是行不通的。两个错误的本质相同——用语言指令解决架构限制。

## 五大设计原则

### 1. 物理约束 > 指令约束

**问题**：告诉 AI"忽略 X"是无效的——LLM 的注意力机制会看到所有在 context 中的内容。

**解决方案**：让不该看到的内容在错误阶段**根本不存在**。

```
❌ 无效：告诉 AI"忽略下一章提示"
✅ 有效：把下一章提示放在 AI 物理上访问不到的地方
```

**实现方式**：
- 文件分离：把"提示"类文件和"创作参考"类文件分开
- 规则声明：在 CLAUDE.md 中显式声明每个阶段"不读什么"
- 工作区隔离：检查点文件存储在独立目录

**效果**：架构约束比指令约束可靠 100 倍。

---

### 2. 减少 AI 的判断空间

**问题**：描述型路由让 AI 自己判断该读什么、自己决定执行顺序。

```
❌ 描述型路由：
"你说写下一章 → 读 当前状态.md + 最近章节梗概 → 执行 X 阶段"
（AI 在生成时自己判断该不该读这些文件）

✅ 事件-动作型路由：
触发：人类说"写下一章"
立即执行：
1. 读取 当前状态.md
2. 读取 最近章节梗概
3. 读取人物档案
4. 生成章纲
5. 停止，等待人类
```

**核心区别**：
- 描述型：AI 在生成时自己判断
- 事件-动作型：触发和动作绑定，AI 必须显式执行

**效果**：减少 AI 的判断空间，增加流程的确定性。

---

### 3. 流程强制 > AI 自觉

**问题**：AI 写完初稿后可以选择跳过自查直接交人类审。

```
❌ 依赖自觉：
写完初稿 → AI 觉得挺好 → 直接交人类审

✅ 流程强制：
写完初稿 → 必须进入自查阶段 → 没得选
```

**核心区别**：把"自查"从"可选步骤"变成"必选步骤"，用流程强制质量保证，而不是依赖 AI 的自觉性。

**效果**：关键步骤不会被跳过。

**实践增强**：自查建议采用“执行层 + 评估层”双层结构，降低单 Agent 漏检。

---

### 4. 记录决策过程 > 只记录结果

**问题**：只记录"做了什么"不记录"为什么这么做"，后人无法理解决策过程。

```
❌ 只记录结果：
{
  "issues_fixed": 3,
  "issues_pending": 0
}

✅ 记录决策过程：
{
  "issues_fixed": 3,
  "revision_details": [
    {
      "issue": "第53行：重复碎片状态确认",
      "type": "S-文风",
      "action": "删除",
      "reason": "违反微调意图学习库案例9：稳定状态不重复提及"
    }
  ],
  "boundary_cases_noted": [
    {
      "line": 59,
      "content": "世界正在变大",
      "decision": "保留",
      "reason": "是秦力真实感受，不是叙述者画重点"
    }
  ]
}
```

**效果**：boundary_cases_noted 是神来之笔——后人能看懂决策过程，不只是结果。

---

### 5. 状态外部化

**问题**：AI 的上下文窗口有限，记忆不可靠。中断后需要人类重新说明上下文。

**解决方案**：将状态**外部化**到文件，作为真相源。

```json
{
  "knowledge_snapshot": {
    "current_date": "正月十一",
    "character_status": {"秦力": "收到学校协查函"},
    "pending_foreshadowing": ["西郊物流园风险"]
  }
}
```

**好处**：
- AI 中断后可以从检查点恢复，无需人类重新说明
- 可以追溯"这问题从哪章开始埋下"
- 可以统计各阶段的创作质量

**效果**：状态文件作为真相源，不应该只存在于 AI 的上下文里。

## 理论基础

### 相关研究

**LangGraph / Microsoft Agent Framework**：
> "A checkpoint captures the entire state of the workflow: current executor state, pending messages, shared states. Enables resume without re-running entire chain."

**MindStudio 研究**：
> "The workflow's state object should be the source of truth, not the agent's context window."
> "Put the workflow layer in charge of sequencing, state, error handling, and routing. Don't ask the agent to manage these — they're deterministic concerns that belong in code."

**Medium (2025)**：
> "Your AI Agent Didn't Fail — It Stopped Halfway."
> "The LLM should not be responsible for tracking what has already been executed, managing checkpoints, or deciding how to handle a mid-workflow failure. These are infrastructure concerns, not reasoning concerns."

**ArXiv 研究**：
> "Agents frequently repeat faulty actions after early-stage failures without updating their internal state, leading to persistent task collapse."

### 核心转变

| 维度 | 旧版思维 | 新版思维 |
|------|----------|----------|
| **目标** | 让 AI 聪明地完成任务 | 让 AI 可靠地执行流程 |
| **约束方式** | 指令约束 | 架构约束 + 流程约束 |
| **AI 角色** | 判断者 | 执行者 |
| **失败处理** | 依赖 AI 自觉 | 用流程强制 + 检查点恢复 |

## 原则在系统中的对应

| 原则 | 在 Craft Companion 中的实现 |
|------|----------------------------|
| **物理约束 > 指令约束** | Context 物理隔离——每个阶段只读必要文件 |
| **减少 AI 的判断空间** | 事件-动作型路由——触发绑定动作序列 |
| **流程强制 > AI 自觉** | 强制自查——写完初稿必须进入自查 |
| **记录决策过程** | boundary_cases_noted + revision_details |
| **状态外部化** | 检查点机制——knowledge_snapshot |

## 总结

Craft Companion 的设计理念可以用一句话概括：

> **"让 AI 可靠地执行流程"而非"让 AI 聪明地完成任务"。**

这是工程化 AI 的正确方向：
- 减少 AI 的判断空间 → 提高可控性
- 物理隔离 context → 提高质量和效率
- 强制执行关键步骤 → 保证质量底线
- 外部化状态 → 支持中断恢复和追溯

这不是"让 AI 更聪明"，而是"让 AI 更可靠"。
