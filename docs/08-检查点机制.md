# 检查点机制

## 什么是检查点

检查点是每阶段完成后的**状态快照**，让 AI 恢复后能从正确位置继续，无需人类重新说明上下文。

## 为什么需要检查点

| 问题 | 检查点解决方案 |
|------|----------------|
| AI 中断后需要重新说明上下文 | 从检查点恢复，无需人类重新说明 |
| 无法追溯"这问题从哪章开始" | 检查点链条可追溯 |
| 无法统计创作质量 | 检查点记录自查问题数、修订次数 |

## 文件结构

### 文件位置

```
工作区/检查点/
└── 第XX章/
    ├── outline_provided_YYYYMMDD_HHMMSS.json
    ├── outline_selected_YYYYMMDD_HHMMSS.json
    ├── draft_completed_YYYYMMDD_HHMMSS.json
    ├── self_review_completed_YYYYMMDD_HHMMSS.json
    ├── revision_completed_YYYYMMDD_HHMMSS.json
    └── final_confirmed_YYYYMMDD_HHMMSS.json
```

### 字段说明

| 字段 | 说明 | 必须性 |
|------|------|--------|
| `checkpoint_version` | 检查点版本，用于后续格式演进 | 必须 |
| `chapter` | 章节号 | 必须 |
| `phase` | 阶段标识（见下表） | 必须 |
| `timestamp` | ISO 8601 时间戳 | 必须 |
| `trigger` | 触发原因 | 必须 |
| `previous_checkpoint` | 指向前一个检查点，形成链条 | 必须（null表示起点） |
| `inputs` | 本阶段读取的文件 | 必须 |
| `inputs.skipped_files` | 本阶段故意没读的文件（用于调试隔离机制） | 建议 |
| `outputs` | 本阶段产出的文件 | 必须 |
| `knowledge_snapshot` | 人物状态、伏笔等上下文快照 | 阶段1必须 |
| `human_confirmed` | 是否经过人类确认 | 关键字段 |
| `notes` | 补充说明 | 可选 |

### 阶段标识

| phase | 含义 | human_confirmed |
|-------|------|-----------------|
| `outline_provided` | 提供了3版章纲 | false |
| `outline_selected` | 人类选定了章纲 | **true** |
| `draft_completed` | 初稿完成 | false |
| `self_review_completed` | 自查完成 | false |
| `revision_completed` | 修订完成 | false |
| `final_confirmed` | 终版确认 | **true** |

## 恢复流程

当 AI 中断后重启时，执行以下步骤：

### 步骤1：检查检查点目录

```
工作区/检查点/第XX章/
├── outline_selected_20260323_070000.json
├── draft_completed_20260323_090000.json    ← 最新
└── self_review_completed_20260323_103000.json
```

### 步骤2：读取最新检查点

读取最新的检查点文件，发现：
- `phase`: "self_review_completed"
- `human_confirmed`: false

### 步骤3：判断下一步

| 状态 | 下一步 |
|------|--------|
| `human_confirmed` = true | 进入下一阶段 |
| `human_confirmed` = false | 等待人类确认 |

### 步骤4：恢复上下文

从 `knowledge_snapshot` 恢复：
- 当前日期
- 人物状态
- 待推进伏笔

## 检查点链条

```
outline_provided → outline_selected → draft_completed → self_review_completed → revision_completed → final_confirmed
       ↓                  ↓                 ↓                   ↓                    ↓                 ↓
   (自动)           (人类确认✅)       (自动)              (自动)                (自动)           (人类确认✅)
```

## 文件命名规范

```
[phase]_YYYYMMDD_HHMMSS.json
```

示例：
- `outline_provided_20260323_070000.json`
- `outline_selected_20260323_070000.json`
- `draft_completed_20260323_090000.json`
- `self_review_completed_20260323_103000.json`
- `revision_completed_20260323_120000.json`
- `final_confirmed_20260323_140000.json`

## 常见问题

### Q: 检查点文件需要手动创建吗？

A: 当前版本是文件约定，AI 在每阶段完成后自动创建。未来可以做成 CLI 工具自动化。

### Q: 可以删除旧的检查点文件吗？

A: 可以，但建议保留以支持追溯。建议的清理策略：
- 终版确认后，保留最近 3 章的检查点
- 更早的检查点可以归档到单独目录

### Q: 如果检查点文件损坏怎么办？

A: 检查点文件是纯 JSON，如果损坏只能从知识库手动重建上下文。

### Q: `skipped_files` 字段有什么用？

A: 显式记录本阶段故意没读的文件。如果 Context 物理隔离机制有 bug，这个字段会暴露问题。

### Q: `boundary_cases_noted` 为什么重要？

A: 记录"未违规但需说明"的边界案例，方便后人理解决策过程，而不只是看到冷冰冰的数字。
