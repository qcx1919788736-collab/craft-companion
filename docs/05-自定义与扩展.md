# 自定义与扩展

NovelForge 是一个灵活的框架，可以根据你的需求定制和扩展。

## 定制知识库结构

### 添加新的文件类型

如果你需要追踪其他类型的信息，可以在知识库中添加新文件。

**示例：添加"场景库"**

```markdown
知识库/
└── 05_场景库/
    ├── 常用地点_完整版.md
    └── 重要物品_完整版.md
```

在 `CLAUDE.md` 中添加说明：

```markdown
| `05_场景库/` | 常用地点、重要物品 | 涉及该场景时读 |
```

### 调整分层结构

根据作品类型调整：

**短篇小说**：
- 可以合并 `00_核心上下文/` 和 `03_故事进展/`
- 可能不需要 `伏笔追踪表`

**长篇系列**：
- `02_世界观设定/` 可能需要更细的分类
- `03_故事进展/` 可能需要分卷

**多视角作品**：
- 为每个主要视角创建独立的"当前状态"
- 文风规则可能需要区分不同视角

## 定制工作流

### 简化工作流

如果你觉得 6 阶段太复杂，可以简化：

**3 阶段简化版**：
1. 章纲（人类提供） → 正文产出 → 归档

跳过章纲发散和 AI 自查，适合经验丰富的作者。

### 增强工作流

如果需要更严格的质量控制：

**8 阶段增强版**：
1. 章纲发散
2. 选择
3. 正文产出
4. AI 自查
5. 人类修正
6. **二次自查**（修正后再检查一遍）
7. **同行评审**（其他作者/读者反馈）
8. 归档

### 自定义阶段

在 `CLAUDE.md` 中修改工作流定义：

```markdown
## 工作流程

### 创作新章节（你的自定义流程）

1. **你的阶段1**：...
2. **你的阶段2**：...
```

## 定制文风规则

### 添加新的规则维度

`文风规则_完整版.md` 可以添加你关心的维度：

```markdown
## 节奏控制

- **信息密度**：每段推进1-2个信息点
- **场景切换**：每章不超过3个场景
- **时间跨度**：单章时间跨度不超过24小时

## 特殊技巧

- **留白**：重要情绪用留白，不要说透
- **对比**：用环境对比衬托人物情绪
```

### 量化你的风格

用具体数字替代模糊描述：

❌ 模糊：
```markdown
- 适当使用短句
- 环境描写不要太多
```

✅ 量化：
```markdown
- 短句（10字以内）每章不超过5处
- 环境描写每个场景不超过2句
```

## 定制错题集

### 添加你的错误模式

`错题集_完整版.md` 应该记录你和 AI 协作中发现的问题：

```markdown
## [你的错误类型]

**错误描述**：[具体描述]

**错误示例**：
> [AI 写的错误版本]

**正确示例**：
> [你修改后的版本]

**修正原则**：[为什么这样改]

**出现章节**：第X章
```

### 分类索引

如果错题集很长，添加分类索引：

```markdown
## 场景类型索引

- **对话场景**：AI-001, AI-003, AI-007
- **战斗场景**：S-001, S-004
- **情感场景**：E-002, E-005
```

## 扩展 CLI 工具

### 添加新命令

编辑 `tools/novel-cli.js`，添加新功能：

```javascript
function yourNewCommand(param) {
  // 你的逻辑
}

// 在 main() 中添加
switch (command) {
  case 'your-command':
    yourNewCommand(param);
    break;
  // ...
}
```

### 自动化知识库更新

当前 `update-kb` 只显示清单，你可以实现自动更新：

```javascript
function autoUpdateKb(chapterNum) {
  // 读取终稿
  // 提取信息
  // 更新知识库文件
}
```

## 集成其他工具

### 版本控制

使用 Git 追踪知识库变化：

```bash
git init
git add 知识库/
git commit -m "第X章完成后的知识库状态"
```

### 备份策略

定期备份知识库：

```bash
# 每完成一章备份
cp -r 知识库/ _归档/知识库备份_第X章/
```

### 导出工具

编写脚本将终稿导出为其他格式：

```javascript
// tools/export.js
// 将工作区/第X章_终版.md 导出为 PDF/EPUB 等
```

## 适配不同 AI 模型

### 针对其他模型优化

如果使用 GPT-4 或其他模型，可能需要调整：

**CLAUDE.md 调整**：
- 简化指令（某些模型不擅长复杂指令）
- 增加示例（某些模型需要更多示例）
- 调整 prompt 结构

**文风规则调整**：
- 某些模型可能需要更明确的禁止项
- 某些模型可能需要更多正面示例

### 模型特定的错题集

为不同模型维护不同的错题集：

```
04_写作参考/
├── 错题集_Claude.md
├── 错题集_GPT4.md
└── 错题集_通用.md
```

## 多人协作

### 分工方案

**方案 1：章节分工**
- 作者 A 负责主线章节
- 作者 B 负责支线章节
- 共享知识库，各自维护自己的错题集

**方案 2：流程分工**
- 作者 A 负责章纲和初稿
- 作者 B 负责修订和归档
- 共享知识库和工作流

### 协作规范

在 `CONTRIBUTING.md` 中定义：

```markdown
## 协作规范

- 修改知识库前先 pull 最新版本
- 每完成一章提交一次
- 冲突时以最新终稿为准
```
