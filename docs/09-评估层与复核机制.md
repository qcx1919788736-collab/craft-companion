# 评估层与复核机制

本文档定义“执行层 + 评估层”的双层质量保障，用来降低单 Agent 自查漏检。

## 为什么要加评估层

单 Agent 在同一上下文里“写作 + 自查”时，容易出现：
- 对自己文本的盲区
- 对边界问题判断过宽
- 把风格偏好误判成硬错误

评估层的目标不是重写正文，而是独立核查执行层结论。

## 双层分工

- 执行层（Writer）：产出正文，完成第一轮自查并给出修复建议
- 评估层（Evaluator）：不参与写作，只审查“执行层报告 + 正文”
- 人类（Arbiter）：只裁决争议项，不重复做全量检查

## 工作顺序（对外仍是 5 阶段）

1. 章纲
2. 初稿
3. 自查（阶段3内部拆为 3.1 执行层自查 + 3.2 评估层复核）
4. 修订
5. 终版确认

## 评估层输入

最小输入包：
- `工作区/第XX章_初稿.md` 或修订稿
- 执行层自查报告
- `知识库/00_核心上下文/当前状态.md`
- `知识库/00_核心上下文/最近章节梗概.md`
- 相关人物档案
- 相关世界观规则
- `知识库/04_写作参考/错题集_完整版.md`

## 评估层输出格式

建议按三分法输出：

- **确认问题（confirmed）**：证据充分，可直接改
- **争议问题（disputed）**：存在解释空间，交人类裁决
- **误报问题（dismissed）**：不成立，说明驳回理由

每条问题必须包含：
- `type`：人物/设定/时间线/文风/叙事效率/伏笔
- `evidence`：原文片段 + 知识库依据
- `suggestion`：具体改法（禁止空泛描述）

## 落地规则

- 无争议问题：直接修改正文，并在报告中记录“改前→改后”
- 有争议问题：在原文对应位置换行追加“建议改写块”，保留原文供人类对比
- 误报问题：必须写明误报原因，避免后续重复报错

## 与检查点联动

建议在 `self_review_completed` 后追加一份 `evaluation_completed` 检查点（可选扩展），记录：
- 评估层确认问题数
- 争议问题数
- 误报问题数
- 是否等待人类裁决

## 最小执行建议

第一版不必上复杂编排器，先做到三点：
1. 每章固定产出“执行层报告 + 评估层报告”两份文件
2. 争议问题必须进入“建议改写块”对比模式
3. 终版前必须清空“未裁决争议”
